## Notes for whoever picks this up

- H1 is genuinely new and reproducible every time: any workflow that needs debug mode ON *and* Depot placement in the same session must remember to toggle debug off before clicking the line/train (the same caveat `CLAUDE.md` already documents for line-drawing — "Draw lines before enabling debug mode, never after" — turns out to apply equally to Depot Train/Carriage assignment, but this isn't written down anywhere yet). Worth deciding whether (a) this is acceptable as "debug mode is a dev tool, know its limits," in which case DEBUG.md's "does not alter game rules" line should be softened/scoped, or (b) `onMouseDown` should check `selectedReserveItem` before falling into the debug-mode branch, so Depot assignment keeps working regardless of debug mode (matching the "at any time" wording in `core/logic.md` §2 Reserve).
- H2 is a code-review-level finding (matches the race condition flagged in the task brief almost exactly) — I could not force it into an observed negative-reserve-count outcome via mouse-only Playwright automation because every scripted attempt's inter-click latency happened to exceed the 10Hz sync window, but the missing guard is real and visible directly in `src/input/mouseHandler.ts`. A human tester clicking very fast by hand, or a differently-timed automation approach (e.g. dispatching raw DOM events without Playwright's per-action actionability waits), would have a better shot at forcing it. Recommended fix if picked up: guard both branches with `> 0` before decrementing, e.g. `if (line && state.reserveCarriers > 0) { addTrainToLine(state, line.id); state.reserveCarriers -= 1; }`.
- Did not get a clean, repeatable confirmation of attaching a **second** Depot Carriage to the same already-carriaged Train (to check 3-box rendering / capacity stacking) — my scripted attempt's train-locating pixel scan returned suspiciously identical coordinates across two supposedly-different moments and the attach silently no-op'd both times (reserve count never decremented across two grants). Given the single-carriage-attach path is confirmed working and time was limited, I did not chase this further; it's worth a dedicated follow-up (ideally with a proper debug-overlay-based train locator rather than a pixel-color scan) rather than trusting my inconclusive script.
- No new camera or line-drawing regressions found; `Extend a Line` / `Mid-Line Insertion` (no dedicated spec files, previously verified free-form in the 2026-07-04 report) were not re-tested this run — out of this run's focus scope.
