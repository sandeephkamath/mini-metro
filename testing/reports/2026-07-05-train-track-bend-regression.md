# Test Report — 2026-07-05 — Train/Track Bend Regression Pass (post `getSegmentElbow` fix in `trains.ts`)

**Focus of this run**: verify the just-fixed regression (trains rendering off their visible line path when a bend gets "mirrored" for overlap-avoidance) did not recur, with extra stress on multi-line bends and shared-station overlaps — the exact bug class involved. Full fixed suite + free-form exploration per `specs/testing.md`.

## Suite result

`cd testing && npm test`: **10 passed, 1 failed** (`weekly-upgrade.spec.ts`). Reproduced identically across two full runs. See "Testing harness notes" below — this failure is a test-design issue (game behavior is correct), not a game bug, so it is intentionally **not** in the findings table.

## Headline answer: did the train/track desync regression recur?

**No.** Across every scenario below — including the specific combinations called out as highest-risk (3+ lines meeting at one station forcing bend-mirroring, mid-line insertion on an already-bent segment, extending a bent line from an endpoint, and a 5-line hub stress test with forced non-aesthetic "kink" bends) — the train marker's on-screen position matched the drawn line path exactly, at rest, mid-segment, mid-bend/elbow, and through both forward and backward (return) trips, confirmed visually via screenshots at each stage. No case of a train floating off its track was found.

### Scenarios exercised specifically for this regression class

1. **3 lines converging on one station from different angles** (circle–triangle, triangle–square, triangle–hexagon, with the hexagon placed to force a bend-conflict at the shared triangle station). Ran ~10 forward/backward cycles at 4x debug speed with screenshots every 1.5-5s. Every train (red/blue/green) stayed glued to its own colored track through the hub, including while stopped at the hub and mid-corner.
2. **5 lines converging on one station** (`triangle` hub, spokes at up-right/up-left/down-right/down-right-shallow/down-left angles specifically chosen to be within `BEND_CONFLICT_ANGLE` (30°) of each other, forcing `getSegmentElbow`'s mirrored/kinked-bend branches), plus 2 extra unlock-filler stations to reach 5 unlocked Lines. 8 screenshots across ~15s of 4x-speed run time (many full round trips per line, since spoke lengths are short). All 5 trains (red/blue/green/orange/purple) tracked correctly through the hub convergence zone in every frame, including while multiple trains occupied the hub simultaneously.
3. **Mid-Line insertion on a bent segment**: drew circle→triangle (one natural diagonal-then-straight bend), then dragged from a point on the segment's straight leg to a new debug-added station, splitting the bend into two new segments. Ran the train through both new segments, forward and backward, across 5 round trips. Train stayed glued through both new bends every time (screenshots `ins-04` through `ins-08`).
4. **Extending an already-bent line from an endpoint**: drew circle→triangle (one bend), then grabbed the triangle end's tab (`getLineEndpointAt`/`endpointHandle`, which also calls `getSegmentElbow`) and extended it to square, producing a 3-station line with two independently-bent segments. Ran ~5 round trips; train tracked correctly through both bends in both directions every time (`ext2-03` through `ext2-06`).

No screenshot in any of these runs showed a train rectangle detached from, or offset from, its line's drawn stroke — confirming the fix (`getSegmentElbow` now passed into every `buildSegmentShape` call in `trains.ts`, matching what `renderLines.ts` draws) is working as intended, including in the exact multi-line-overlap conditions that caused the original regression.

Not a bug, just a visually confusing (by construction of my own test, not a real player scenario) but spec-correct outcome.

## Testing harness notes (not game bugs — flagging for whoever maintains `testing/`)

1. **`weekly-upgrade.spec.ts` fails because it never draws a Line.** The test waits 95s of wall time (≈380s of game time at 4x speed) for the 5-week/300s Milestone Event, but never draws any Line to relieve spawning Passengers. With 3 starting Stations and no way to deliver anyone, a Station's queue reaching its 6-Passenger capacity and then sitting there past the 8s Risk Timer is essentially inevitable well before 300 game-seconds elapse — confirmed live: the test's own screenshot/page-snapshot on failure shows a normal "Game Over — A station overflowed" screen, i.e. the game is behaving correctly per `core/logic.md` §3 Node Overflow; the test's assumptions are just stale after the Weekly Upgrade interval was widened from 60s to 300s in commit `f8d75c5` (the wait time was updated but the test still doesn't draw a relieving Line). Recommended fix: draw at least one Line connecting two starting Stations before the long wait, and/or pause passenger auto-spawn (`P`) for stations not on that Line, so the intended assertion (week ≥ 5) actually gets a chance to run.
2. **`testing/helpers/gameDriver.ts`'s local `ALL_SHAPES` constant is stale.** It lists only `['circle', 'triangle', 'square']`, but the game's real shape list (`src/logic/shapes.ts`) has 6: `circle, triangle, square, star, hexagon, plus`. Per `specs/DEBUG.md`, the "Add Station" popup always shows **all 6** shapes (including locked ones), and the "Add Passenger" popup shows every shape that differs from the station's own shape (up to 5, not 2). Because `debugAddStation`'s and `debugAddPassenger`'s button-position math (`clampMenu`, index arithmetic) is derived from the stale 3-item list, `debugAddStation` is currently unusable as-is (every attempted call in my first script silently missed the shape button and placed nothing) — I only got it working by reimplementing the button math locally in a throwaway spec with the real 6-shape list. `debugAddPassenger` happens to still work in the current fixed suite only because the specific stations/destinations used there don't hit the cases where the stale count changes the computed button position or index. Recommended fix: update `ALL_SHAPES` in `gameDriver.ts` to match `src/logic/shapes.ts`'s 6 shapes (or import shape metadata from a shared source), since this will only get more likely to bite as more flows use debug station/passenger injection near canvas edges or with non-circle/triangle/square destinations.

## Method notes

- All free-form multi-line/bend scenarios were driven via `testing/run-driver.mjs` (interactive REPL) and one throwaway `testing/flows/tmp-*.spec.ts` file per scenario (deleted after use, per isolation rules — none remain in the repo).
- Placing a debug station outside the Camera's current auto-fit margin (`core/logic.md` §5) triggers an immediate re-center/zoom-out, which silently invalidates any hardcoded screen-space coordinates for subsequent actions in the same script — this cost significant time to diagnose (looked like "drawing a line did nothing" when it was actually "the drag landed on empty panned-away space and started a camera pan instead"). Worked around by either keeping all added stations within the camera's default safe viewing box, or issuing a zero-delta wheel event (`onWheel` with `deltaY=0`) early to permanently disable auto-fit (any manual zoom/pan does this per `core/logic.md` §5) so subsequent screen coordinates stay valid for the rest of the script regardless of where stations are added.
