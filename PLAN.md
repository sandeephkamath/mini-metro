## Sequence

- [x] **1. Persistence + Weeks Survived core** (`meta_progression.md` §1, §2, §6)
  Single localStorage key. Read once on load (before home screen), write once at session end (Best Weeks Survived + Collectible progress). Silent fail if storage unavailable.
  Done: `src/logic/metaProgression.ts` (`getWeekProgress`/`getWeeksSurvived`), `src/storage/metaProgression.ts` (`loadMetaProgression`/`recordSessionEnd`), wired into `src/hooks/useGameState.ts` (seeds `bestWeeksSurvived` on mount, records once per session on the `playing` → `gameover` transition). Verified end-to-end via the run-driver: forced a station overflow in debug mode, confirmed `localStorage['miniMetro.metaProgression.v1']` updates with a nonzero `bestWeeksSurvived` after Game Over. UI display (game-over/home screen text) is deferred to phase 4 — this phase only builds the tracking + storage layer, not code implementing the display. `collectionSize`/`currentPictureProgress` fields are reserved in the stored shape but stay at 0 until phase 2/3.

- [x] **2. Collectible Reward progression (logic only)** (`meta_progression.md` §3)
  Accumulated Progress, Required Progress curve (`base × growthRate^(N-1)`), Reveal Steps, Minimum Session Contribution floor, Complete → Collection, surplus carry-forward. No rendering yet.
  Done: `src/logic/collectibles.ts` (`getRequiredProgress`, `getRevealStepProgress`, `getRevealedTileCount`, `applySessionContribution`), fallback config added to `gameConfig.ts` (`PICTURE_BASE_REQUIREMENT: 20`, `PICTURE_GROWTH_RATE: 1.5`, `PICTURE_TILE_COUNT: 20`, per `metro.md` §9.3's table). Wired into `storage/metaProgression.ts`'s `recordSessionEnd`, replacing the phase-1 pass-through. Verified end-to-end by driving 21 consecutive sessions: `currentPictureProgress` climbed 1→19 via the Minimum Session Contribution floor, session 20 hit `required(1)=20` exactly and rolled to `collectionSize:1`/`currentPictureProgress:0`, and session 21 correctly used Picture 2's own Reveal Step (`required(2)=30 → step 1.5`) landing at `1.5` — matches spec math exactly, including completion and surplus carry-forward. No Firestore/curated-pool override support yet (that's phase 3's `metro.md` §9.3.1) and no rendering.

- [x] **3. Picture presentation, fallback pool** (`metro.md` §9.3, §9.4)
  Procedurally rendered Pictures from a small built-in fallback dataset (1-2 cities), using existing line/station shape-drawing code. Animated Game-Over Reveal. Firestore-backed content fetch deferred — build/test against fallback pool per memo.md's note.
  Done: `src/data/pictureCities.ts` (2 hand-authored fallback cities — simplified London Underground and Paris Métro, real line colors, shared interchange stations), `src/logic/pictureContent.ts` (`getPictureForIndex`, mod-wraparound beyond the curated pool per §9.3.1), `src/render/renderPicture.ts` (`renderPictureFull` reusing `computeBentSegment`/`traceShapePath` — the same bend-geometry and station-shape primitives live gameplay uses — plus `drawRevealedPicture` for tile compositing). Extended `src/logic/collectibles.ts` with `RevealSegment`/`getSessionRevealSegments` (per-Picture start/end progress spans for a session, refactoring `applySessionContribution` on top without changing its behavior), and `storage/metaProgression.ts`'s `recordSessionEnd` now returns `{ data, segments }`. New `src/components/PictureReveal.tsx` animates the count-up + tile pop-in (`PICTURE_REVEAL_ANIM_MS` config) and "Picture Complete!" celebration through however many Pictures one session spans, wired into `GameOverScreen.tsx`/`GameCanvas.tsx`/`useGameState.ts` (`pictureRevealSegments`).
  Verified end-to-end: regression-ran the phase-2 21-session driver script (identical results, confirming the segment refactor didn't change behavior), then visually confirmed via screenshots — a single-tile 5% reveal, the full London map at 97%/99%/100% with tiles popping in correctly, the "Picture Complete!" celebration at 100%, and Picture 2 (Paris) starting fresh at 5%/"+1.5 weeks" after rollover. No Firestore integration (still fallback-pool only) and no Home Screen/Collectibles Screen UI yet — that's phase 4.

- [x] **4. Home Screen + Game Over wiring** (`home_screen.md`, `metro.md` §9.1/§9.2)
  Best Weeks Survived line (omitted if 0), Picture thumbnail, "View Collectibles" control, Collectibles Screen modal (Complete / Current / Up-next locked).
  Done: `src/components/PictureThumbnail.tsx` (static reveal-state canvas, shared by Home Screen + Collectibles Screen), `src/components/CollectiblesScreen.tsx` (Complete oldest-first, Current at live reveal state, 3 locked `"???"` upcoming placeholders + "...and more", per home_screen.md's own "generic placeholder is the safe default" guidance). `renderPicture.ts` grew a shared `getPictureCanvas` cache (also adopted by `PictureReveal.tsx`, removing its per-component cache). `HomeScreen.tsx` now takes `bestWeeksSurvived`/`collectionSize`/`currentPictureProgress` props, shows "Best: Week N" (omitted at 0) and the Picture thumbnail + "View Collectibles" button below the Play control, and owns local `showCollectibles` modal state. `GameOverScreen.tsx` shows "You survived to Week N — Personal Best is Week M" or a distinct "New Best! Week N" callout (metro.md §9.2) — `storage/metaProgression.ts`'s `recordSessionEnd` now also returns `isNewBest`. `useGameState.ts` consolidated the scalar `bestWeeksSurvived` into a full `metaProgression: MetaProgressionData` state (still seeded once on mount, updated once per session end) plus a new `isNewBest` flag, both reset on `startGame`/`goHome`.
  Verified end-to-end via the driver: fresh home screen (no Best line, blank Picture thumbnail, Collectibles modal showing 0 complete + locked lookahead), a forced-overflow session showing "New Best! Week 0", then the home screen afterward correctly showing "Best: Week 0" and a 1-tile-revealed thumbnail, matching in both the Home Screen and Collectibles Screen. No Leaderboard control (phase 6, not yet implemented) — correctly absent per the "fail gracefully" rule, not a gap.

- [x] **5. Monetization** (`core/monetization.md`, `metro.md` §4.2)
  Simulated Ad Provider stand-in → On-Demand Bonus Request → Game-Over Continue (Continue Limit, rescues Overflow Risk stations, Final Weeks Survived only read on real end). Also remove the old `'grace'` milestone bonus kind (`src/logic/milestone.ts` `AUTO_ORDER` and `grantBonus`) since Risk Timer duration is now fixed for the session.
  Spec fix first: `core/logic.md` §3 Milestone Events still said "all three kinds" for Choice mode, contradicting its own "exactly one of two kinds" a few lines up (and `progression.md`/`monetization.md`, already correct) — a stale leftover from the earlier monetization spec pass. Fixed before touching code.
  Done: removed `'grace'` from `MilestoneBonusKind` (`types/game.ts`), `AUTO_ORDER`, and `RISK_TIMER_INCREMENT_MS`; `milestone.ts`'s bonus-granting logic extracted into exported `grantReserveBonus` (shared by the free Milestone Event and both ad-gated paths). Old `MilestoneChoiceModal` replaced by generic `BonusChoiceModal` (`title`/`subtitle` props) so the Weekly Upgrade and the ad-flow bonus pick reuse the identical presentation per spec. New `src/logic/monetization.ts` (`isAdAvailable`, `canOfferContinue`, `requestOnDemandBonus`, `offerGameOverContinue`, `accept`/`decline`/`completeAdPlayback`, `resolveAdBonusChoice`) drives a single `state.adFlow: AdFlowState | null` (`kind: 'onDemand'|'continue'`, `stage: 'confirm'|'playing'|'choice'`) — `gameLoop.ts`'s `tick()` freezes on it exactly like `milestoneChoicePending`. `overflow.ts`'s Grace Timer expiry now calls `offerGameOverContinue` instead of ending the game directly when `canOfferContinue` is true. New `AdConfirmModal`/`SimulatedAdModal` components (`CONFIG.SIMULATED_AD_DURATION_MS`, wall-clock driven since the Game Clock is already frozen). HUD grew a "🎬 Free Bonus" button, shown only while `adAvailable`. `DEBUG.md`'s `V` toggle wired into `useMouseInput.ts` (`playing` phase only, resets when debug mode turns off). `continuesRemaining` (`CONFIG.CONTINUE_LIMIT`) resets every session, decremented only on a completed Continue.
  Caught and fixed one bug before shipping: `gameLoop.ts` only early-returned on `phase === 'gameover'` after `updateOverflowRisk` — since offering a Continue doesn't flip `phase`, week/milestone advancement would have snuck through in the same tick a Continue was offered. Added an explicit `if (state.adFlow) return` there too.
  Verified end-to-end via the driver, all 4 paths: On-Demand Bonus (confirm → ad → choice → Depot Train granted, button stays available after), Debug Ad Availability toggle (button hides/reappears with `V`), Game-Over Continue accepted (offer → ad → choice → station queue trimmed to `maxCapacity-1` + Risk Timer cleared + session resumes, day-of-week clock kept advancing), Continue Limit exhaustion (a second overflow in the same session skips the offer entirely and ends the game, with Final Weeks Survived/Best/Picture-reveal only recorded at that real end — not the earlier rescued one), and Continue declined (ends the game immediately). No real ad SDK — Simulated Ad only, as spec'd for this stage.
